# Auditoría SEO BogleHub — 21 mayo 2026

> Auditoría completa pre-implementación. No se ha tocado ningún código. Estado real, no aspiracional.

---

## Resumen ejecutivo

BogleHub tiene una base técnica sólida (Next.js 16 App Router, Vercel, HTTPS, mobile-first) y contenido bien escrito con intención correcta. Sin embargo, hay **cuatro problemas estructurales** que están lastrando la indexación antes de empezar. Dos de ellos son P0 y se resuelven en días.

### Los 5 hallazgos críticos

| # | Hallazgo | Impacto | Prioridad |
|---|----------|---------|-----------|
| 1 | **Cero structured data en artículos del blog** | Google no puede leer autor, fecha ni publisher; E-E-A-T invisible | P0 |
| 2 | **`/analyzer`, `/comparar` y calculadoras son Client Components puros** | Googlebot ve un esqueleto HTML vacío en las páginas de producto | P0 |
| 3 | **Title double-BogleHub en 4 páginas** | Títulos como "... \| BogleHub \| BogleHub" penalizan CTR | P0 |
| 4 | **Título del blog `/blog` es solo "Blog"** | Sin keyword targeting en el hub de contenido | P1 |
| 5 | **15 artículos publicados en 9 días sin dateModified** | Señal de AI content burst; sin historial de actualizaciones visible | P1 |

Lo que está bien y hay que mantener: `<html lang="es">`, `metadataBase`, FAQ schema en home, WebApplication/Organization en home, HTTPS, Core Web Vitals baseline excelente, Server Components en blog y ETF, disclaimers YMYL presentes.

## A.1 — Auditoría técnica

### Crawlability

**robots.ts** — Correcto.
```
Allow: /
Disallow: /api/
Sitemap: https://boglehub.vercel.app/sitemap.xml
```
Ningún problema. La API no necesita indexarse.

**sitemap.ts** — Cubre: rutas estáticas, /etf/[ticker] (68), /blog/[slug] (15). Total estimado: ~90 URLs.

Problemas detectados:
- `/score` (`src/app/score/page.tsx` existe) **no está en el sitemap**. P1.
- Prioridades inconsistentes: `/aviso-legal`, `/privacidad`, `/terminos` tienen `priority: 0.8` — igual que el analizador. Las páginas legales deberían estar en `0.3`. P2.
- `changeFrequency: 'weekly'` para ETF pages — los datos de ETF no cambian semanalmente. `'monthly'` es más honesto. P2.
- `lastModified: new Date()` para todas las rutas — esto le dice a Google que TODO cambia en cada deploy. Mejor usar fechas reales. P2.

### Core Web Vitals

No puedo ejecutar Lighthouse sin CLI, pero la arquitectura predice un rendimiento excelente:
- Next.js 16 + Turbopack: SSR por defecto en Server Components.
- Vercel Edge CDN: latencia baja desde España.
- Geist font con `next/font`: sin layout shift de fuentes.
- `next/image` implícito: lazy loading y WebP automáticos si se usa correctamente.

**Riesgo real**: El glob 3D (`GlobeHero`) en el hero de home es un Three.js bundle. Si no está lazy-loaded con suspense, puede afectar LCP en mobile. Revisar con Lighthouse real tras Phase B.

### Metadata global

**layout.tsx** — Template: `'%s | BogleHub'`. Default: `'BogleHub'`. 

**BUG CRÍTICO (P0):** Las siguientes páginas incluyen "| BogleHub" dentro del string del título, y además reciben el sufijo del template, resultando en doble BogleHub:

| Página | Título configurado | Título real renderizado |
|--------|--------------------|------------------------|
| `/` (homepage) | `'BogleHub — Análisis de cartera para inversores indexados'` | `'BogleHub — Análisis de cartera para inversores indexados \| BogleHub'` |
| `/calculadora/interes-compuesto` | `'Calculadora de interés compuesto España 2026 \| BogleHub'` | `'...\| BogleHub \| BogleHub'` |
| `/calculadora/fire-monte-carlo` | `'Calculadora Monte Carlo para tu jubilación FIRE \| BogleHub'` | `'...\| BogleHub \| BogleHub'` |
| `/comparar` | `'Comparar ETFs España 2026 \| BogleHub'` | `'...\| BogleHub \| BogleHub'` |

Fix: usar `title: { absolute: '...' }` en estas páginas, o eliminar el "| BogleHub" del string y dejar que el template lo añada.

### Render: Server vs Client

| Ruta | Render | Impacto SEO |
|------|--------|-------------|
| `/blog/[slug]` | **Server Component** ✓ | Contenido completo indexable |
| `/etf/[ticker]` | **Server Component** ✓ | Contenido completo indexable |
| `/` (home) | **Server Component** ✓ | OK |
| `/metodologia` | **Server Component** ✓ | OK |
| `/blog` | **Server Component** ✓ | OK |
| `/analyzer` | **`'use client'` en `page.tsx`** ✗ | H1, cuerpo, tabla de posiciones: invisibles para Googlebot |
| `/comparar` | **`EtfComparator.tsx` es `'use client'`** ✗ | Todo el comparador invisible |
| `/calculadora/interes-compuesto` | **`CompoundInterestCalculator.tsx` es `'use client'`** ✗ | Invisible |
| `/calculadora/fire-monte-carlo` | **`MonteCarloCalculator.tsx` es `'use client'`** ✗ | Invisible |
| `/chat` | Client via `Chat` component ✗ | Invisible |

**Diagnóstico**: El metadata (title, description) de las páginas herramienta se propaga desde los `layout.tsx` / `page.tsx` del servidor — Google LAS INDEXA con el título/descripción correcto. Pero el cuerpo de la página (H1, contenido, formularios, texto explicativo) está en el Client Component. Googlebot puede leer:
- El `<title>` correcto ✓
- El `<meta description>` correcto ✓
- El H1 del JSX servidor (si existe) — en `/analyzer/page.tsx` el H1 `<h1>Analizador de cartera</h1>` SÍ está en el Client Component → Googlebot NO lo ve.

Impacto: estas páginas no son candidatas a ser ricas en contenido SEO de todas formas (son herramientas interactivas), pero perder el H1 y cualquier texto descriptivo es una señal negativa. La solución no es reescribir la herramienta, sino añadir un wrapper de servidor con H1 y descripción visible.

### HTTP Status / Canonical

- `/cartera` tiene `robots: { index: false }` — correcto (página de cartera compartida, sin valor SEO standalone).
- No hay canonical explícito definido en ninguna página. Next.js genera un canonical implícito basado en la URL. Para rutas sin parámetros duplicados, esto es suficiente. Sin embargo, para ETF pages si existen variantes case-sensitive (`/etf/VWCE` vs `/etf/vwce`) hay riesgo de duplicado. Verificar que `generateStaticParams` solo genera lowercase. ✓ (confirmado: `ticker.toLowerCase()`).

### Hallazgos de calidad

**Lo bueno:**
- Todos los artículos revisados tienen una estructura H1 → H2 → H3 coherente.
- El contenido de los artículos 1, 2, 3 es genuinamente profundo: tablas de datos reales, comparativas concretas, ejemplos numéricos verificables.

- Las keywords objetivo son razonables: no exact-match spam, natural language.
- Slugs en español, limpios, sin stop-words innecesarias.

**Lo problemático:**
- **Burst de publicación**: 15 artículos en 9 días es una señal de alerta para Google HCU. No implica penalización automática, pero refuerza la necesidad de E-E-A-T sólido visible (autor, actualizaciones, fuentes).

- **Slug con año hardcoded**: `como-elegir-tu-primer-etf-espana-2026` — en 2027 parecerá desactualizado. Solución: actualizar el contenido y añadir `dateModified`, no cambiar la URL (perderías el posicionamiento).
- **`fondos-indexados-vs-etfs-espana` y `solapamiento-etfs-error-silencioso` comparten fecha**: 15 de mayo. Dos artículos el mismo día no es problema, pero es otra señal del burst.

## A.5 — Auditoría AI Search

**Metodología**: evaluación estructural (no puedo verificar directamente en Perplexity/ChatGPT sin acceso en tiempo real, pero sí puedo evaluar si el contenido cumple los criterios de citabilidad).

### Criterios de citabilidad para LLMs

| Criterio | Estado BogleHub |
|----------|----------------|
| Definición clara en los primeros 2 párrafos | ✓ Los artículos revisados abren con contexto claro |
| Datos cuantitativos verificables (tablas) | ✓ Artículos 1, 2, 3 tienen tablas excelentes |
| Estructura pregunta → respuesta directa | ⚠️ Parcial. Los H2 son afirmativos, no preguntas |
| FAQPage schema | ✓ Solo en home. ❌ En artículos: ninguno |
| `<details>/<summary>` para FAQ | ✓ En home. ❌ En artículos del blog |
| Fuentes citadas explícitas | ⚠️ Los artículos mencionan datos (TER, AUM) pero sin links a fuentes primarias (Vanguard, iShares, JustETF) |
| Autoridad de dominio | ❌ Sitio nuevo, sin backlinks conocidos |
| Contenido "citation-worthy" único | ✓ El grado fiscal (A-F) por ETF es ÚNICO en español — diferenciador real |

### Hipótesis sobre citabilidad actual

BogleHub **no aparece** en AI Overviews ni Perplexity para queries del nicho en este momento. Razones probables:
1. El sitio tiene menos de 2 semanas de contenido indexado.
2. Sin backlinks, la autoridad de dominio es mínima.
3. Las páginas más completas (blog) sí tienen buena estructura de citabilidad, pero aún no han acumulado señales de confianza.

El potencial es real: los artículos están escritos con el formato que los LLMs prefieren extraer (tablas, definiciones claras, "Si X entonces Y"). El trabajo es construir autoridad de dominio (backlinks, señales sociales) y completar el schema.
